DOCUMENT-DOMAINS-001 — Dynamic Document Domains & Access Control

STEP 0 — SAFETY FIRST

Before making any changes:

1. Create Replit checkpoint/snapshot.
2. Create git tag if possible:

v3.7-before-dynamic-document-domains

3. Confirm build passes.
4. Confirm current Documents, Domains, Roles, Degrees, Dashboard and History still work.

Only then start implementation.

--------------------------------------------------
GOAL
--------------------------------------------------

Redesign document access so it is configurable, dynamic, and reusable for other lodges.

Do not hardcode document access rules in application logic.

Access must be based on configurable domain rules using:
- Roles
- Degrees
- Role/Degree logic

The system must automatically adapt when a user's role or degree changes.

--------------------------------------------------
CORE PRINCIPLE
--------------------------------------------------

Access must be calculated dynamically at runtime.

Do NOT permanently copy folder access onto individual users.

Examples:

If a user receives Treasurer role:
- Treasury Documents immediately become visible.

If Treasurer role is removed:
- Treasury Documents immediately disappear.

If an Entered Apprentice becomes Fellowcraft:
- Fellowcraft Ritual immediately becomes visible.

If a Fellowcraft becomes Master Mason:
- Master Mason Ritual immediately becomes visible.

No manual folder grants should be required.

--------------------------------------------------
DOMAINS REDESIGN
--------------------------------------------------

Rename or evolve:

Management → Domains

into:

Management → Domains & Access Control

The Domains module should manage access rules for protected areas.

Each domain must support:

1. Domain name
2. Domain key
3. Description
4. Allowed roles
5. Allowed degrees
6. Access logic

Access logic options:

- Role Only
- Degree Only
- Role OR Degree
- Role AND Degree

Optional later:
- Explicit user overrides
- Temporary grants
- Expiry dates

Do not implement overrides in this sprint unless already simple.

--------------------------------------------------
DOCUMENT MANAGEMENT
--------------------------------------------------

Normal Documents page is for viewing allowed folders only.

All management must be under:

Management → Document Management

Visible only to:
- Site Administrator
- PM Super Administrator

Document Management allows:
- Create main folder if permitted
- Create subfolder
- Rename folder
- Delete empty folder
- Edit folder description
- Link folder to a Domain
- Move folder between General Documents and Ritual Documents frame if needed

Subfolders inherit parent folder domain/access by default.

No custom subfolder permissions in this sprint unless already easy.

--------------------------------------------------
SITE ADMIN RULE
--------------------------------------------------

Site Administrator is a technical administration role.

Site Administrator may manage structure through Document Management.

Site Administrator does NOT automatically get read access to restricted content.

Specifically, Site Administrator should not automatically see:
- Secretary Documents
- Treasury Documents
- Worshipful Master Documents
- Past Master Ritual

unless that same user also has the required lodge role or degree.

--------------------------------------------------
PM SUPER ADMIN RULE
--------------------------------------------------

PM Super Administrator may be included in sensitive domains as emergency governance access.

Default:
- PM Super Administrator can access restricted administrative folders.
- PM Super Administrator can access ritual folders only where included by the domain rule.

For Pershing default setup, include PM Super Administrator in all default restricted domains.

--------------------------------------------------
FOLDER VISIBILITY
--------------------------------------------------

If a user has no access to a folder:
- Hide the folder completely.

Do not show:
- disabled folder
- locked folder
- access denied message

Backend must also deny access.

Frontend hiding is not enough.

--------------------------------------------------
DATABASE / MODEL
--------------------------------------------------

Implement or extend data model so domains can store:

- allowedRoleIds
- allowedDegreeIds
- accessLogic
- description
- domainKey

Folders should reference a domain/access rule.

Avoid hardcoding access decisions in React components.

Access should be checked server-side.
